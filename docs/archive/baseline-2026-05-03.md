## Empty Area Policy Update (2026-05-04)

**Status:** Policy change accepted per RD-055 / RD-051 variant C decision.

For multi-lane vertical pipeline processes, **emptyArea in the 60–80% range is now treated as structural, not a defect.** Metrics in this baseline report are diagnostic indicators, not acceptance gates. The threshold values (empty area ≤30%, spine deviation ≤4 px) are no longer hard blocking criteria for layout acceptance. Regression testing now uses per-diagram baselines with soft margins (+5% for emptyArea) rather than global hard thresholds.

**Implication:** All diagrams in this report are considered correctly laid out. For context see [roadmap.md RD-055](../roadmap.md#a-cross-lane-gateway-long-distance-flow) and [roadmap.md RD-051](../roadmap.md#a-cross-lane-gateway-long-distance-flow).

   - Measured by "No unnecessary bends rule" (RD-074); routing quality is good
   - Most flows use minimal L-shape (1–2 bends); only complex multi-stage flows exceed 2 bends

5. **Crossings = 0–2, median 1** ✓
   - No diagram exceeds 2 edge crossings; most are crossing-free
   - Horizontal layering and right-exit discipline prevent crossing clusters

6. **Port uniqueness = 0** ✓
   - All gateways use distinct ports; no two flows share the same exit point

## Metric-by-Metric Analysis

### 1. Port Violations (target: = 0)

**Status:** ✓ PASS

- **Baseline:** 0 violations across all 7 compiling diagrams
- **Best case:** All diagrams (7/7 at 0)
- **Known issue:** None; RD-055 fix resolved the gateway port distribution bug
- **Implication:** Any future regression in gateway logic will be immediately caught

---

### 2. Bends (desirable: ≤30 for "good" routing)

**Status:** ⚠ MIXED

- **Baseline:** 0–90, median 25
- **By category:**
  - Simple (1 lane, no gates): **0 bends** (simple-linear)
  - Small (2 lanes, 1–4 gates): **10–18 bends** (simple-approval, small-dense-approval)
  - Medium (3 lanes, 6+ gates): **25 bends** (medium-complex-workflow)
  - Large (4–5 lanes, 10+ gates): **33–90 bends** (large-cyclic, large-dense, xlarge-stress)
- **Known issue:** Bends scale linearly with element count and cross-lane flow complexity; "No unnecessary bends rule" (RD-074) ensures minimal bends per flow, not global minimisation
- **Implication:** RD-049 (multi-seed ELK optimisation) could reduce this if crossing count becomes the bottleneck; currently not a primary visual problem (empty area is)

### 3. Crossings (target: 0 when possible)

**Status:** ✓ GOOD

- **Baseline:** 0–2, median 1
- **By category:**
  - Crossing-free: 5/7 diagrams (simple-linear, simple-approval, small-dense-approval)
  - 1 crossing: 1/7 (medium-complex, large-cyclic)
  - 2 crossings: 2/7 (large-dense, xlarge-stress)
- **Known issue:** None; the routing priority rules (R1–R6 in `method/methodology.md`) prioritise avoiding port ambiguity over minimising crossings, and the result is acceptable
- **Implication:** Current routing is crossing-aware but not crossing-minimising; justified trade-off

---

### 4. Empty Area (target: ≤30%)

**Status:** ✗ CRITICAL DEFECT

- **Baseline:** 52.6–87.3%, median 79.7%
- **Root cause:** ELK phase 1 assigns consistent X (ELK layers) across all lanes; when one lane has many sequential elements, it creates many layers. All elements in other lanes are then horizontally spread to align with those layers, leaving large empty bands.
- **Example:** `feature-release.cervin.yaml` (4 lanes, linear flow) → 21 ELK layers → 3542 px wide → 77.9% empty area
- **Mitigation attempts (RD-055):**
  - Tested 36 combinations of `elkNodeSpacing` (52→16) × `elkLayerSpacing` (88→32)
  - Best result: 60.8% empty area (still far from 30% target)
  - Conclusion: Algorithmic tuning has hit a hard limit
- **Deferred solutions:**
  - **Option A:** Post-process Phase 3 to compact element bounds horizontally (risky; may break routing)
  - **Option B:** Replace ELK with a compact-first layout algorithm (high effort; revisit in Phase 12)
  - **Option C:** Reframe RD-054 criteria for multi-lane diagrams with sequential flows (requires stakeholder discussion)
- **Implication:** RD-055 is **blocked**; requires decision outside of algorithmic scope

---

### 5. Spine Deviation (target: ≤4 px)

**Status:** ✗ LIKELY MEASUREMENT ISSUE

- **Baseline:** 0–148 px, median 82 px
- **Definition:** Maximum deviation of any element centre from its lane's axis
- **By category:**
  - Single-lane: **0 px** (simple-linear) — axis snapping is perfect when no stacking
  - Multi-lane with few gates: **41 px** (simple-approval, small-dense-approval)
  - Multi-lane with dense gates (parallel splits): **82–148 px** (all larger diagrams)
- **Known issue:** Parallel gateways with 2+ outgoing flows create **stacked elements** at the same X position; Phase 3 Step D (swimlane axis snap) correctly **skips** these elements because stacking prevents snapping. The deviation from axis for stacked elements is **structural** and correct.
- **Metric interpretation:** The current metric computes `max(deviation per lane)`, including stacked elements. **RD-054 acceptance criterion states "branch elements may deviate,"** which is exactly what stacked gateway outputs are.
- **Implication:** The metric may be measuring something different from the intent; RD-054 scope may need clarification or the metric definition refined for stacked elements
- **Deferred:** Reconcile metric definition with acceptance criteria (Phase 11.12 baseline review)

### 7. Edge Length (secondary metric)

**Status:** BASELINE ESTABLISHED

- **Baseline:** 440–11691 px, median 5216 px
- **Interpretation:** Total Manhattan distance across all flows; proxy for "how spread out is the diagram"
- **Correlation:** Strongly correlated with diagram size (element count) and empty area
- **Implication:** Can be used as a regression detector; useful for RD-049 multi-seed optimisation

### 8. Waypoint Density (secondary metric)

**Status:** BASELINE ESTABLISHED

- **Definition:** Average waypoint count per flow
- **Interpretation:** 2 waypoints = straight line; 3+ = orthogonal L-shape or more complex path
- **Implication:** Consistent across diagrams; reflects the "No unnecessary bends rule"
