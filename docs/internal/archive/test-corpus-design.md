# Test Corpus Design

**Version:** 0.1  
**Date:** 2026-05-03  
**Scope:** Structured collection of BPMN diagrams for regression testing, baseline metrics, and layout algorithm validation.  
**Related:** [`roadmap.md`](../roadmap.md) Phase 11 RD-080–082, [`metrics.md`](metrics.md)

---

## Overview

The test corpus is a stratified sample of BPMN diagrams covering multiple dimensions of complexity. Each diagram is a self-contained `*.bpmn.transitrix.yaml` file with a header comment identifying its coverage cell (new fixtures use this extension only; existing legacy-extension regression fixtures are not migrated). The corpus enables:

1. **Baseline metrics** — compute reference values for each metric across diverse diagram shapes.
2. **Regression testing** — catch performance/quality regressions when layout algorithm changes.
3. **Optimization validation** — judge whether an optimization (e.g., RD-055 column compaction) generalizes or overfits.
4. **Stress testing** — verify scalability on larger diagrams.

## Coverage Dimensions

### Dimension 1: Diagram Size (Element Count)

| Category | Range | Label |
|----------|-------|-------|
| Small | 5–10 | S |
| Medium | 11–25 | M |
| Large | 26–50 | L |
| Extra-large | 51+ | XL |

### Dimension 2: Cross-Lane Share

Fraction of flows connecting elements in different lanes.

| Category | Range | Label |
|----------|-------|-------|
| Low | < 20% | Lo |
| Mid | 20–50% | Mi |
| High | > 50% | Hi |

### Dimension 3: Cycle Presence

Presence of backward flows (flow-to-element where target.id appears earlier in topological sort).

| Category | Present | Label |
|----------|---------|-------|
| Acyclic | No | A |
| Cyclic | Yes | C |

### Dimension 4: Gateway Density

Fraction of elements that are gateways (XOR, AND).

| Category | Range | Label |
|----------|-------|-------|
| Low | < 15% | Lo |
| High | ≥ 15% | Hi |

### Dimension 5: Lane Count

Total number of swimlanes in the diagram.

| Category | Range | Label |
|----------|-------|-------|
| Single | 1 | 1 |
| Two | 2 | 2 |
| Three | 3 | 3 |
| Multi | 4+ | 4+ |

## Coverage Matrix

A complete coverage matrix is **5 dimensions × (4 + 3 + 2 + 2 + 4) = 120 cells**. A stratified sample covers the most relevant cells:

**High-priority cells** (at least one diagram per cell):

| Size | Cross-lane | Cycle | Gateway | Lanes | Rationale |
|------|-----------|-------|---------|-------|-----------|
| S | Lo | A | Lo | 1 | Minimal diagram; baseline |
| S | Mi | A | Lo | 2 | Small with cross-lane |
| S | Hi | A | Hi | 2 | Small, dense gates, cross-lane |
| M | Lo | A | Lo | 1 | Medium, same-lane only |
| M | Mi | A | Lo | 2 | Medium, some cross-lane |
| M | Hi | A | Hi | 3 | Medium, high gates, many cross-lane |
| M | Mi | C | Lo | 2 | Medium, cycle present |
| L | Mi | A | Lo | 3 | Large, sparse gates |
| L | Hi | A | Hi | 4 | Large, dense gates, many cross-lane |
| L | Hi | C | Hi | 4 | Large, cycle, complex |
| XL | Hi | A | Hi | 4+ | Stress test: largest |

**Minimum corpus:** 11 diagrams (one per high-priority cell).  
**Target corpus:** 14–18 diagrams (add 3–7 additional cells for secondary coverage).

## Diagram Authoring Guidelines

### Header Comment

Each `*.bpmn.transitrix.yaml` file begins with a structured comment:

```yaml
# CORPUS CELL: S-Mi-A-Lo-2
# Element count: 8 (5 tasks, 2 events, 1 XOR)
# Cross-lane flows: 2/3 (67%)
# Cycles: none
# Gateway density: 12.5%
# Lanes: 2 (admin, requester)
# Purpose: [brief description of what this diagram tests]
# Source: [synthetic | adapted from X | minimal example]

pool: MyPool
  # ... rest of YAML
```

### Synthetic Diagrams (Preferred)

Design diagrams to **isolate a specific feature**:
- **S-Mi-A-Lo**: simple order→approve→ship across 2 lanes; tests cross-lane routing.
- **M-Hi-C-Hi**: approval loop with parallel gates and backward edge; tests cycles and port distribution.
- **L-Hi-A-Hi-4+**: wide grid of gates and tasks across many lanes; tests scalability.

Avoid unnecessary complexity; each diagram should have a clear purpose.

### Adapted Diagrams (Secondary)

If sourcing from external examples (e.g., open BPMN repositories under compatible license):
1. Note the source and license in the header.
2. Simplify to fit the target corpus cell if too large.
3. Verify the diagram is valid per [`method/methodology.md`](../method/methodology.md) Section 7.

## Notes

- **Corpus is immutable post-baseline.** Once RD-088 baseline is established, corpus files should not be edited (to preserve historical metrics). New diagrams go into a separate `examples/bpmn/corpus-v2/` if major schema changes occur.
- **Round-trip validity.** Every corpus file is tested via `BpmnModdle.fromXML` in CI to ensure valid BPMN 2.0 output.
- **Scaling path.** If corpus proves too small for real-world use cases, create `examples/bpmn/corpus-large/` with 100+ element diagrams (stress test only, not regression baseline).
